# Fabric 1.20.2 专用服启动修复设计

**问题**：使用 `-Pmc_ver=1.20.2` 启动 Fabric 专用服时，Fabric API 的 `fabric-registry-sync-v0` 在引导阶段对 `MappedRegistry` 应用 `SimpleRegistryMixin` 失败。日志报错目标为缺失的 Yarn 字段 `field_26683`，服务端在注册表初始化前终止。

## 根因

`versionProperties/1.20.2.properties` 将 `cloth_config_version` 设为 `13.0.138`。这是 1.20.4 的 Cloth Config 工件；其传递依赖带入 Fabric API `0.91.2+1.20.4`。该版本在解析依赖时覆盖项目显式声明的 `fabric_api_version=0.91.6+1.20.2`，导致 1.20.2 的游戏字节码与 1.20.4 的 Fabric Registry Sync mixin 混用。

日志中出现的 `fabric-registry-sync-v0` `SimpleRegistryMixin` / `field_26683` 注入失败是这一混用的直接证据。Hassium 不向 `MappedRegistry` 注入；Hassium 的服务端入口也尚未执行，故不修改业务代码或 mixin。

## 已比较的方案

1. **采用 Cloth Config `12.0.137`：选定。** Modrinth 将该 Fabric 工件明确标记为支持 Minecraft 1.20.2。它保留开发运行时的 Cloth Config 与 Mod Menu 配置界面，且仅改版本属性。
2. 移除 Cloth Config 和 Mod Menu 的 `modLocalRuntime`：可规避问题，但开发客户端不能自动测试可选配置 UI；不采用。
3. 以细粒度 Fabric API 模块代替聚合 `fabric-api`：需要同时处理 Cloth Config、Mod Menu 的传递依赖，扩大维护面；不采用。

## 设计

只修改 `versionProperties/1.20.2.properties` 的 `cloth_config_version`：

```properties
cloth_config_version=12.0.137
```

`fabric/build.gradle` 已将该属性用于 `modCompileOnly` 和 `modLocalRuntime`，无需新增条件分支、排除规则或运行时代码。发布元数据中 Cloth Config 本来就是 `suggests`，因此不会将其强制为用户运行依赖。

## 不变量

- `fabric_api_version` 保持 `0.91.6+1.20.2`。
- `modmenu_version=8.0.1` 保持不变。
- 其他 Minecraft 版本继续各自使用现有 Cloth Config 版本。
- 修复不更改 Hassium 的网络、存储、Mixin 或配置逻辑。

## 验证

1. 编译 1.20.2 Fabric：`./gradlew --no-daemon fabric:compileJava "-Pmc_ver=1.20.2"`。
2. 启动 `fabric:runServer -Pmc_ver=1.20.2`，观察服务端日志到达 `Done (...)!`，且不出现 `SimpleRegistryMixin`、`InvalidMixinException` 或 `field_26683`。
3. 终止已启动的测试服务。
